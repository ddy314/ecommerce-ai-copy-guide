## 1. 系统总览

### 1.1 项目定位

本项目是一个面向电商场景的智能导购系统，核心能力包括：
- **商品数据采集**：从苏宁易购爬取真实商品数据（1874条商品）
- **AI文案生成**：基于商品信息生成营销文案、直播脚本、选购指南
- **智能问答导购**：用户用自然语言提问，系统检索商品并调用大模型生成回答
- **商家管理后台**：商品管理、知识库管理、订单管理、数据分析

### 1.2 技术栈

| 层级 | 技术选型 |
|------|---------|
| 前端 | Vue 3 + TypeScript + Vite + 原生 CSS |
| 后端 | Flask + SQLAlchemy + PostgreSQL |
| AI 大模型 | DeepSeek-V3（硅基流动 API） |
| Embedding 模型 | BAAI/bge-large-zh-v1.5（硅基流动 API） |
| 向量数据库 | ChromaDB（本地持久化） |
| 爬虫 | requests + BeautifulSoup4 |
| 缓存 | Redis |

### 2.5 京东爬虫失败原因分析

项目最初尝试爬取京东（JD）商品数据，但最终失败，转而使用苏宁易购作为数据源。以下是京东爬取失败的五个主要原因：

**原因一：动态渲染页面，BeautifulSoup 无法解析**

京东搜索结果页大量使用 JavaScript 动态渲染商品列表。商品数据不在初始 HTML 中，而是通过 AJAX 异步加载后由前端框架渲染。项目的爬虫使用 `requests + BeautifulSoup` 技术栈，只能获取静态 HTML，无法执行 JavaScript：

```python
# 京东搜索URL
url = f"https://search.jd.com/Search?keyword={keyword}&page={page_param}"

# requests 获取的是未执行JS的初始HTML
html = requests.get(url).text
soup = BeautifulSoup(html, "parser")

# 尝试多种选择器，全部找不到商品节点
items = soup.select("li.gl-item")        # 旧版结构 → 0个结果
items = soup.select("div.gl-i-wrap")     # 新版结构 → 0个结果
items = soup.select("[data-sku]")        # 其他选择器 → 0个结果
# 根本原因：商品数据在JS渲染后才出现在DOM中
```

相比之下，苏宁易购的搜索页是服务端渲染（SSR），商品数据直接嵌入在初始 HTML 中，BeautifulSoup 可以直接解析。

**原因二：反爬机制严密，频繁触发验证码**

京东拥有国内最严格的反爬体系之一，包括：
- 请求频率检测：短时间内多次请求同一搜索接口会触发风控
- Cookie/Session 验证：京东要求完整的 Cookie 链路，简单请求会被拒绝
- 滑块验证码：触发风控后会重定向到验证页面

```python
# 代码中的验证码检测逻辑
if "验证" in resp.text[:500] or "login" in resp.url:
    logger.warning("可能触发京东验证，尝试下一页")
    return None
elif resp.status_code == 302:
    logger.warning("302重定向，可能需要验证")
    return None
```

实际运行中，爬取前2-3页后就会触发验证码，导致后续页面全部获取失败。苏宁的反爬措施相对宽松，正常请求频率下不会触发验证。

**原因三：代理IP质量不足，被京东识别封禁**

项目配置了快代理隧道代理（`proxy_config.py`），但京东的IP信誉库比苏宁更完善：

```python
# 快代理隧道代理配置
PROXY_HOST = "c126.kdlttps.com"
PROXY_PORT = "15818"
PROXY_USERNAME = "t18391813951910"
PROXY_PASSWORD = "gadt1qmp"

# 代理请求
resp = self.session.get(url, proxies=get_proxies(), timeout=15)
```

快代理的隧道IP虽然是动态轮换的，但这些IP段已被京东标记为"数据中心IP"或"爬虫IP"，京东会直接拒绝这些IP的请求或返回降级页面。而苏宁对代理IP的检测较弱，同样的代理可以正常访问。

**原因四：京东页面结构频繁变动，选择器失效**

京东前端采用组件化架构，页面结构迭代频繁。爬虫代码中尝试了三种CSS选择器来兼容不同版本：

```python
# 尝试旧版结构
items = soup.select("li.gl-item")
# 尝试新版结构
items = soup.select("div.gl-i-wrap")
# 尝试其他结构
items = soup.select("[data-sku]")
```

即使页面能正常加载，京东的HTML结构变更也会导致选择器全部失效，需要频繁维护代码。苏宁的页面结构相对稳定，选择器维护成本低。

**原因五：京东商品价格异步加载，搜索页无法直接获取**

京东搜索页的价格通过单独的AJAX接口动态加载，不在商品HTML节点中：

```python
# 京东价格解析逻辑
price_elem = item.select_one("div.p-price i")
# 实际上 div.p-price 内的价格是JS填充的，初始HTML中为空或为占位符
# → price_elem 为 None → 价格解析失败
```

京东的价格接口需要携带商品SKU和Token请求 `https://p.3.cn/prices/mgets?skuIds=J_SKUID`，且该接口也有反爬检测。苏宁的价格直接写在HTML中，BeautifulSoup可以直接提取。

**总结与方案对比：**

| 维度 | 京东 (JD) | 苏宁易购 |
|------|----------|---------|
| 页面渲染 | JavaScript动态渲染（CSR） | 服务端渲染（SSR） |
| 反爬强度 | 极高（验证码+Cookie+IP检测） | 中等（UA检测即可绕过） |
| 代理IP兼容性 | 差（数据中心IP被识别） | 好（正常代理可用） |
| 页面结构稳定性 | 频繁变动 | 相对稳定 |
| 价格获取 | 需额外AJAX请求 | 直接在HTML中 |
| 爬取成功率 | <10% | >95% |

最终选择苏宁易购作为数据源，成功爬取了1874条商品数据，覆盖7大分类。

### 4.5 PostgreSQL 数据库

**核心文件：** `backend/database.py`

#### 为什么选择 PostgreSQL

PostgreSQL 是本项目的核心关系型数据库，选择它而非 MySQL/SQLite 的原因：

| 维度 | PostgreSQL | MySQL | SQLite |
|------|-----------|-------|--------|
| ILIKE 模糊搜索 | 原生支持（不区分大小写中文搜索） | LIKE（区分大小写配置复杂） | LIKE（功能受限） |
| JSON 字段 | 原生 JSONB 类型，支持索引查询 | JSON 类型（5.7+） | TEXT 存储 |
| 全文搜索 | 内置中文全文搜索支持 | 需第三方插件 | 不支持 |
| 并发性能 | MVCC 多版本并发控制，高并发读写优秀 | 行级锁，高并发下性能一般 | 单写入者，不适合并发 |
| 数据完整性 | 严格的 ACID 合规 | ACID 合规 | ACID 合规但无网络访问 |

#### 连接配置与降级机制

`database.py` 实现了自动降级策略，确保项目在不同环境下都能运行：

```python
def _resolve_database_url() -> str:
    """解析数据库 URL，PostgreSQL 不可用时降级到 SQLite"""
    configured_url = os.getenv("DATABASE_URL", "")

    # 情况1：未配置 → 使用 SQLite
    if not configured_url:
        return "sqlite:///./ecommerce_ai.db"

    # 情况2：配置了 PostgreSQL → 检查连接可用性
    if configured_url.startswith("postgresql"):
        try:
            import psycopg2  # 检查驱动是否安装
            engine = create_engine(configured_url, pool_pre_ping=True)
            with engine.connect() as conn:
                conn.execute(text("SELECT 1"))  # 测试连接
            return configured_url  # PostgreSQL 可用
        except ImportError:
            return "sqlite:///./ecommerce_ai.db"  # psycopg2 未安装 → 降级
        except Exception:
            return "sqlite:///./ecommerce_ai.db"  # 连接失败 → 降级

    return configured_url
```

**降级链路：**
```
PostgreSQL 连接成功 → 使用 PostgreSQL（生产环境，pool_size=10）
    ↓ 如果失败
psycopg2 未安装 / 连接超时 → 降级到 SQLite（开发环境，单文件）
```

#### 连接池配置

生产环境（PostgreSQL）使用连接池优化数据库性能：

```python
engine = create_engine(
    DATABASE_URL,
    pool_size=10,          # 连接池保持10个常连接
    max_overflow=20,       # 高峰时可额外创建20个连接（最大30）
    pool_pre_ping=True,    # 每次取连接前检测是否存活（防止使用已断开的连接）
    echo=True,             # 开发环境打印SQL语句（生产环境设为False）
)
```

- **pool_size=10**：常驻10个数据库连接，避免频繁创建/销毁连接的开销
- **max_overflow=20**：高并发场景下（如多个用户同时问答）可临时扩展到30个连接
- **pool_pre_ping=True**：PostgreSQL 连接可能因网络波动断开，取用前先发 `SELECT 1` 检测，如果断开则自动重建

### 4.6 Redis 缓存层

**核心文件：** `backend/cache.py`

#### 为什么需要 Redis

在电商场景中，商品数据、AI生成的文案、推荐结果等数据具有以下特征：
- **读多写少**：商品信息被频繁查询但很少修改
- **计算成本高**：AI文案生成、评论情感分析需要调用大模型API
- **重复请求多**：多个用户可能查询同一商品、生成同类型文案

Redis 作为内存数据库，将这些高频访问的数据缓存在内存中，将响应时间从**百毫秒级（数据库查询）** 降到**亚毫秒级（内存读取）**。

#### 缓存架构

```
用户请求 → 检查 Redis 缓存
              │
              ├── 命中（Cache Hit）→ 直接返回缓存数据（<1ms）
              │
              └── 未命中（Cache Miss）→ 查询 PostgreSQL / 调用 AI API
                                         │
                                         └── 结果写入 Redis（设置TTL过期）
```

#### 降级机制

Redis 不可用时系统自动降级，不影响核心功能：

```python
class CacheService:
    def __init__(self, default_ttl: int = 3600):
        self.default_ttl = default_ttl  # 默认缓存1小时
        self._available = None          # None=未检测, True=可用, False=不可用

    @property
    def client(self) -> redis.Redis | None:
        if self._available is False:
            return None  # 已知不可用，直接跳过
        try:
            client = get_redis()
            if self._available is None:
                client.ping()           # 首次使用时检测连接
                self._available = True
            return client
        except redis.ConnectionError:
            self._available = False     # 标记为不可用，后续不再尝试
            return None                 # 返回None，调用方走数据库
```

**降级链路：**
```
Redis 连接成功 → 使用 Redis 缓存（亚毫秒级响应）
    ↓ 如果连接失败
标记 _available=False → 所有缓存操作跳过 → 直接查询 PostgreSQL（百毫秒级）
```

#### 缓存键命名规范

系统为不同类型的数据定义了规范的缓存键：

```python
# 商品详情缓存
product_key(product_id)    → "product:123"

# 商品评论缓存
reviews_key(product_id)    → "reviews:123"

# AI文案缓存（按商品+语气风格组合）
copy_key(product_id, tone) → "copy:123:professional"

# 推荐结果缓存（按用户需求哈希）
recommend_key(need_hash)   → "recommend:a3f8b2c1"

# 情感分析结果缓存
sentiment_key(product_id)  → "sentiment:123"
```

#### 缓存策略

| 缓存对象 | TTL | 策略 | 原因 |
|---------|-----|------|------|
| 商品详情 | 1小时 | 主动失效 | 商品信息可能被商家修改 |
| 商品评论 | 30分钟 | 短TTL | 评论会持续新增 |
| AI文案 | 24小时 | 长TTL | 生成成本高，同一商品+语气组合结果固定 |
| 推荐结果 | 10分钟 | 短TTL | 用户需求可能变化 |
| 情感分析 | 1小时 | 中TTL | 评论数据变化后需重新分析 |

#### 数据序列化

Redis 存储的是 JSON 序列化后的字符串，支持复杂对象：

```python
def set(self, key: str, value: Any, ttl: int | None = None) -> None:
    serialized = json.dumps(value, ensure_ascii=False)  # 支持中文
    self.client.setex(key, ttl or self.default_ttl, serialized)

def get(self, key: str) -> Any | None:
    data = self.client.get(key)
    if data is None:
        return None
    return json.loads(data)  # 自动反序列化
```

#### PostgreSQL 与 Redis 的协作关系

```
┌─────────────────────────────────────────────────────────┐
│                     数据访问层                            │
│                                                         │
│  用户请求 → CacheService.get(key)                        │
│              │                                          │
│              ├── 命中 → 返回缓存数据（<1ms）              │
│              │                                          │
│              └── 未命中 ↓                                │
│                                                         │
│  SessionLocal() → PostgreSQL 查询（~50ms）               │
│              │                                          │
│              └── 结果 → CacheService.set(key, data, ttl) │
│                         │                                │
│                         └── 写入 Redis 缓存              │
│                                                         │
│  下次相同请求 → 命中缓存（<1ms）                          │
└─────────────────────────────────────────────────────────┘

PostgreSQL: 持久化存储，数据不会丢失，但查询需要磁盘IO
Redis:     内存存储，查询极快，但重启后数据丢失（可作为缓存）
ChromaDB:  向量存储，专用于余弦相似度搜索
```

## 7. 双阶段混合检索策略

### 7.1 为什么需要双阶段

单一的检索方式各有致命缺陷：

| 检索方式 | 优势 | 致命缺陷 |
|---------|------|---------|
| 纯规则检索 | 数值约束精确、可解释 | 语义理解差，"0-6月"匹配不到"1段" |
| 纯向量检索 | 语义理解强、跨语言 | 数值约束失效，¥581和¥220在向量空间几乎等距 |

**双阶段策略取两者之长：** 用规则保证约束精确性，用向量保证语义排序质量。

### 7.2 第一阶段：规则过滤

目标：从1874条商品中快速筛选出满足硬约束的候选集。

```
用户问题: "推荐一款300元以内的蓝牙耳机"

Step 0: 精确商品名匹配
  ├── 用户直接粘贴商品名时，优先精确匹配
  └── 检查价格约束是否满足

Step 1: 核心关键词搜索 (Primary Keywords)
  ├── 提取核心关键词: ["蓝牙", "耳机"] (来自CATEGORY_KEYWORDS映射)
  ├── SQL: WHERE name ILIKE '%蓝牙%' AND name ILIKE '%耳机%'
  ├── 应用价格约束: price <= 300
  ├── 应用规格约束: 无
  └── 候选集 A (如: 15个商品)

Step 2: 通用关键词搜索 (如果Step 1无结果)
  ├── 扩展为通用关键词: ["蓝牙", "耳机", "无线", "降噪"]
  ├── SQL: WHERE name ILIKE '%蓝牙%' OR name ILIKE '%耳机%'
  ├── 应用所有约束
  └── 候选集 B

Step 3: 全局搜索 (如果Step 2无结果)
  ├── 获取分类下所有商品
  ├── 应用所有约束
  └── 候选集 C

Step 4: 分类降级 (如果Step 3无结果)
  └── 移除关键词约束，只保留价格+规格约束
```

**规则过滤的关键约束：**

| 约束类型 | 提取方式 | 过滤逻辑 |
|---------|---------|---------|
| 价格上限 | 正则 `(\d+)元以内` | `price <= limit` |
| 价格范围 | 正则 `(\d+)元左右` | `price >= limit*0.7 AND price <= limit*1.3` |
| 分类 | 关键词→分类映射表 | `category = detected_category` |
| 人群 | 关键词→人群映射表 | 商品名包含人群关键词 |
| 年龄段 | 年龄→段位映射表 | 商品名包含段位关键词 |

### 7.3 第二阶段：向量重排序

目标：对候选集做语义排序，选出与用户意图最匹配的商品。

```
候选集 (15个商品，已通过规则过滤)
    │
    ▼
┌──────────────────────────────────────────────┐
│ 1. 构造查询文本                                │
│    query_text = "推荐一款300元以内的蓝牙耳机"   │
│              + " " + "蓝牙 耳机" (关键词增强)   │
│                                              │
│ 2. 调用硅基流动API生成查询Embedding             │
│    POST https://api.siliconflow.cn/v1/embeddings │
│    model: BAAI/bge-large-zh-v1.5             │
│    → 1024维向量                               │
│                                              │
│ 3. ChromaDB向量搜索                            │
│    collection.query(                          │
│        query_embeddings=[查询向量],             │
│        where={"product_id": {"$in": 候选ID}},   │
│        n_results=15                           │
│    )                                          │
│    → 返回每个商品的余弦距离                     │
│                                              │
│ 4. 距离→相似度转换                             │
│    similarity = 1 - distance / 2             │
│    (0=完全相同, 1=不相关)                      │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────┐
│ 5. 综合评分排序                                │
│                                              │
│ 综合评分 = 向量相似度 × 0.6                    │
│         + 关键词匹配度 × 0.3                   │
│         + 商品评分   × 0.1                    │
│                                              │
│ 示例:                                         │
│ 商品A: 向量0.85×0.6 + 关键词25×0.3 + 4.7×0.1  │
│      = 0.51 + 0.75 + 0.47 = 1.73             │
│ 商品B: 向量0.72×0.6 + 关键词15×0.3 + 4.5×0.1  │
│      = 0.43 + 0.45 + 0.45 = 1.33             │
│                                              │
│ → 商品A得分更高，选为最佳推荐                   │
└──────────────────────────────────────────────┘
```

### 7.4 降级机制

系统设计了完整的降级链路，确保任何环节失败都不影响用户体验：

```
向量索引可用？
  ├── 是 → 向量重排序（最佳质量）
  └── 否 → 关键词加权排序（降级，仍可用）

AI大模型可用？
  ├── 是 → DeepSeek-V3生成自然语言回答
  └── 否 → 模板回答（降级，格式化输出）
```

### 7.5 双阶段 vs 单阶段对比

| 场景 | 纯向量检索 | 纯规则检索 | 双阶段混合检索 |
|------|-----------|-----------|--------------|
| "300元以内的耳机" | 可能返回¥581的耳机（语义相似但价格超限） | 返回¥220耳机✓ | 返回¥220耳机✓ |
| "0-6月婴儿奶粉" | 可能返回12-36月奶粉（语义相似） | 需手动映射0-6月→1段 | 规则过滤+向量排序✓ |
| "适合老人的手机" | 可能返回任意手机（不理解"老人"约束） | 匹配商品名含"老人"的手机 | 规则过滤+向量排序✓ |
| "蓝牙耳机推荐" | 返回语义最相似的耳机✓ | 返回ilike匹配的耳机 | 规则过滤+向量排序✓ |

## 8. 两大知识来源

本系统的知识来自两个独立的数据源，分别服务于不同的检索场景。

### 8.1 知识来源一：PostgreSQL 商品数据库

**用途：** 商品检索、AI问答的主数据源

**数据表：**

```
products (商品表 - 1874条)
├── id              主键
├── name            商品名称（如"EDIFIER/漫步者 X3 AIR 蓝牙耳机..."）
├── category        商品分类（如"数码电子"）
├── price           商品价格
├── brand           品牌
├── selling_points  卖点描述
├── specs           规格参数（JSON）
├── image_url       商品图片
├── rating          评分（1-5分）
├── review_count    评价数量
└── sales_count     销量

reviews (评论表)
├── product_id      关联商品
├── rating          评分
├── content         评论内容
└── created_at      评论时间

shopping (购物车/订单/收藏)
├── Cart            购物车
├── Order           订单
└── Favorite        收藏
```

**检索方式：** SQL ilike + 正则约束 + 向量重排序（双阶段混合检索）

**特点：**
- 数据来自爬虫实时采集
- 结构化存储，支持精确数值过滤
- 通过ChromaDB建立向量索引，支持语义搜索
- 是AI问答的核心数据源

### 8.2 知识来源二：知识库表 (knowledge_entries)

**用途：** 商品规格、FAQ、售后政策的结构化知识存储

**数据表：**

```
knowledge_entries (知识库条目表)
├── id              主键
├── product_id      关联商品ID（可为空，空=通用知识）
├── category        知识类型: spec(规格) / faq(常见问题) / after_sale(售后)
├── title           条目标题
├── content         知识内容
├── keywords        关键词列表(JSON)
├── vector_data     简化向量数据(JSON)
└── is_active       是否启用
```

**知识类型：**

| 类型 | 说明 | 示例 |
|------|------|------|
| `spec` | 商品规格参数 | "电池续航：24小时；防水等级：IP55" |
| `faq` | 常见问题 | "这款耳机支持同时连接两台设备吗？" |
| `after_sale` | 售后政策 | "支持7天无理由退换，质保1年" |

**检索方式：** 关键词加权匹配（非向量检索）

```
用户问题 → 提取关键词 → 遍历知识库条目 → 加权打分 → 返回Top5

打分规则:
  关键词在内容中出现:  +0.3/个
  关键词在keywords中出现: +0.5/个
  关键词在标题中出现:   +0.2/个
  完整问题在内容中出现:  +0.5
```

**自动构建：** 商家添加商品后，系统自动从商品数据生成知识条目：
- 从 `product.specs` 生成 `spec` 类型知识
- 从 `product.selling_points` 生成 `faq` 类型知识

### 8.3 两个知识源对比

| 维度 | PostgreSQL商品数据库 | 知识库表 |
|------|---------------------|---------|
| 数据来源 | 爬虫自动采集 | 商家手动管理 + 自动生成 |
| 检索方式 | 双阶段混合检索（规则+向量） | 关键词加权匹配 |
| 数据类型 | 商品基础信息（名称/价格/评分） | 深度知识（规格/FAQ/售后） |
| 向量索引 | 有（ChromaDB 1024维） | 无（纯关键词） |
| 用途 | 找商品、推荐商品 | 查规格、查FAQ、查售后 |
| 更新频率 | 随爬虫更新 | 商家随时管理 |

## 9. 流式输出机制

### 9.1 流式输出全链路

```
用户输入问题
    │
    ▼ 前端发送POST请求
┌──────────────────────────────────────────┐
│ 前端 (UserLayout.vue)                     │
│ fetch('/api/user/qa/stream', {            │
│   method: 'POST',                         │
│   body: JSON.stringify({question}),       │
│ })                                        │
│ → 获取 ReadableStream reader              │
│ → 逐行解析 SSE data: {...}                │
│ → 通过 chatMessages.value[msgIdx] 更新     │
│ → Vue响应式触发逐字渲染                    │
└──────────────────┬───────────────────────┘
                   │ HTTP Stream
                   ▼
┌──────────────────────────────────────────┐
│ 后端 (user_routes.py: qa_stream)           │
│ Response(generate(), mimetype="text/event-stream") │
│                                          │
│ generate() 是一个生成器函数:               │
│ for event in rag_service.answer_question_stream(): │
│     yield f"data: {json.dumps(event)}\n\n" │
└──────────────────┬───────────────────────┘
                   │ Python Generator
                   ▼
┌──────────────────────────────────────────┐
│ RAG服务 (rag_service.py: answer_question_stream) │
│                                          │
│ yield {"type": "product", "data": {...}}  │ ← 先推送商品卡片
│ yield {"type": "related", "data": [...]}  │ ← 推送推荐商品
│                                          │
│ for chunk in llm_stream:                  │ ← LLM流式输出
│     yield {"type": "text", "content": chunk} │
│                                          │
│ yield {"type": "done"}                    │ ← 完成信号
└──────────────────┬───────────────────────┘
                   │ OpenAI SDK stream=True
                   ▼
┌──────────────────────────────────────────┐
│ 硅基流动 API (DeepSeek-V3)                 │
│                                          │
│ POST /v1/chat/completions                │
│ { "stream": true, ... }                  │
│                                          │
│ → 逐token返回:                            │
│   data: {"choices":[{"delta":{"content":"给"}}]}  │
│   data: {"choices":[{"delta":{"content":"你"}}]}  │
│   data: {"choices":[{"delta":{"content":"推"}}]}  │
│   ...                                     │
└──────────────────────────────────────────┘
```

### 9.2 SSE 事件格式

后端向前端推送的SSE事件有4种类型：

```jsonl
// 1. 商品卡片（回答开始前推送）
data: {"type": "product", "data": {"id": 1, "name": "漫步者X3 AIR", "price": 220, "rating": 4.7, "image_url": "..."}}

// 2. 推荐商品（商品卡片后推送）
data: {"type": "related", "data": [{"id": 2, "name": "漫步者TWS1", "price": 140, "rating": 4.8}, ...]}

// 3. 文本流（LLM逐token推送，多次发送）
data: {"type": "text", "content": "给"}
data: {"type": "text", "content": "你"}
data: {"type": "text", "content": "推荐"}
data: {"type": "text", "content": "漫步者X3 AIR"}

// 4. 完成信号
data: {"type": "done"}
```

### 9.3 前端响应式处理

Vue 3 中流式更新的关键：通过数组索引访问响应式代理对象，确保每次 token 追加都触发视图更新：

```typescript
// 创建空消息
chatMessages.value.push({ role: 'assistant', content: '', product: null, relatedProducts: [] })
const msgIdx = chatMessages.value.length - 1  // 关键：通过索引访问代理

// SSE处理
if (event.type === 'text') {
    chatMessages.value[msgIdx].content += event.content  // 通过代理更新 → 触发响应式
}
```

**打字指示器：** 当 `content` 为空且 `aiSending` 为 true 时，在气泡位置显示三点动画；第一个 token 到达后，`content` 非空，打字指示器自动消失，文本开始逐字显示。

## 10. 向量索引服务

### 10.1 架构

```
backend/services/vector_index.py (VectorIndex 单例)
    │
    ├── Embedding: 硅基流动 API (BAAI/bge-large-zh-v1.5, 1024维)
    │
    ├── 向量数据库: ChromaDB (本地持久化, chroma_db/目录)
    │
    └── 集合: products (1874条记录, cosine相似度)
```

### 10.2 索引构建

```python
# 构建流程
products = SELECT * FROM products  # 从PostgreSQL加载全部商品
for product in products:
    document = f"{name} | {selling_points} | {category} | {brand}"
    embedding = siliconflow_api.embed(document)  # 1024维向量
    chromadb.collection.upsert(id, document, embedding, metadata)

# 结果: 1874条商品全部建立向量索引
```

### 10.3 查询流程

```python
# 双阶段检索的第二阶段
query_embedding = siliconflow_api.embed(user_question)  # 1024维查询向量
results = chromadb.collection.query(
    query_embeddings=[query_embedding],
    where={"product_id": {"$in": candidate_ids}},  # 只在候选集内搜索
    n_results=top_k,
)
# 返回: [{product_id, similarity, ...}, ...]
```

### 10.4 API 接口

| 接口 | 方法 | 用途 |
|------|------|------|
| `/api/vector/build` | POST | 构建/更新向量索引（商家调用） |
| `/api/vector/status` | GET | 查询向量索引状态 |

### 10.5 模型选择说明

**BAAI/bge-large-zh-v1.5：**
- 中文优化的大型Embedding模型
- 输出1024维向量
- 在中文语义相似度任务上表现优秀
- 通过硅基流动API调用，无需本地部署

**ChromaDB：**
- 轻量级向量数据库，无需独立服务
- 本地持久化存储，重启不丢失
- 支持余弦相似度(cosine)、欧氏距离(l2)、内积(ip)
- 支持metadata过滤（`where`条件），实现候选集内搜索
